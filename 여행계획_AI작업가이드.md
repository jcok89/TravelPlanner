# 🧭 AI로 "빈틈없는 여행 계획서" 만들기 — 스킬 & 프롬프트 가이드

> 이 문서는 **동유럽 3국 계획서**(`202608_동유럽3국/index.html`)를 만들 때 실제 사용한 스킬과 프롬프트를 재사용 가능하게 정리한 것입니다.
> Claude Code, Codex, Gemini CLI 등 **웹검색 + 파일작성이 되는 AI 에이전트면 어디서든 재현 가능**합니다.
> 결과물 예시: 날짜별 탭 + 타임라인 + 지도 + 구간별 길찾기 + 예약 우선순위 + 검증된 후기 링크 29개 + 날짜별 예약가능 확인된 투어 추천.

## 0. 한눈에 보는 워크플로우

```
① 확정 정보 고정  →  ② 3-트랙 병렬 리서치  →  ③ 계획서 HTML 제작
(항공/기차/숙소)      (패키지/후기/운영정보)      (날짜탭+지도+길찾기)
        ↓                                              ↓
⑥ git 커밋/푸시  ←  ⑤ 투어 예약가능 조회  ←  ④ 후기 링크 실접속 검증
   + research/ 기록     (마이리얼트립 MCP)        (네이버 블로그)
```

**5대 원칙**
1. **확정 정보(항공·기차·숙소·확정 예약)는 절대 불변** — AI가 임의로 시간/역/주소를 바꾸지 못하게 명시
2. **교차검증** — 여행사 패키지(업계 검증) × 실제 후기(만족도) × 공식 사이트(우리 날짜 운영 여부) 3개가 일치해야 확정
3. **링크는 실접속 검증** — 제목만 보고 넣지 않기 (실제로 4/15개가 제목과 내용 불일치였음)
4. **날짜별 예약 가능 여부까지 확인** — "좋은 상품"이어도 내 날짜에 안 되면 무의미 (실제로 평점 4.9 투어가 우리 날짜 예약 불가였음)
5. **모든 조사는 MD로 기록 + git 푸시** — 조사일 명시, 나중에 재검증 가능하게

## 3. STEP 2 — 3-트랙 병렬 리서치 프롬프트 (복붙용)

서브에이전트를 지원하는 도구면 **3개를 동시에 실행**하세요. `{ }` 부분만 바꾸면 됩니다.

### 트랙 A: 여행사 패키지 분석

```
한국 메이저 여행사 하나투어와 모두투어의 "{지역명}" 패키지 상품을 웹에서 조사해줘.

조사 목표:
1. 두 여행사가 각 도시에서 공통 포함하는 관광 포인트 목록 (일정표 기준)
2. "선택관광(옵션)" 항목과 대략적인 가격
3. 패키지가 방문하는 근교 도시와 소요시간
4. 상품 후기에서 "가장 좋았던 곳" / "아쉬웠던 곳"

여행사 사이트가 동적 페이지라 안 열리면 공식 블로그나 검색 스니펫,
유사 패키지(삼호관광 등) 일정표로 대체해도 좋아.

최종 출력: 도시별 표(관광포인트|A사|B사|기본/선택|가격),
근교도시 옵션, 만족 TOP5/불만 TOP3, 출처 URL 목록.
```

### 트랙 B: 실제 후기 검증

```
{날짜}에 {도시1}({N}박) → {도시2}({N}박) 자유여행을 계획 중이야.
{여행 스타일: 예) 커플 여행, 야경/미식 중심}.
실제 여행자 후기(네이버 블로그, 트립어드바이저, 레딧 등)를 조사해서
도시별 "반드시 가야 할 곳"을 검증해줘.

조사 목표 (도시별):
1. 후기에서 압도적으로 많이 언급되는 필수 스팟 TOP 7 + 추천 방문 시간대
2. "가봤는데 별로였다 / 시간 아깝다"는 스팟
3. 계절 팁: 일몰 시각, 더위/추위, 혼잡 시간대, 소매치기 주의 구역
4. {스타일}에 맞는 극찬 경험 5선
5. 한국인 인기 맛집 최신 후기 — 웨이팅/예약 팁 포함

기준 스팟 목록 (검증 대상): {알고 있는 후보 스팟 나열}

최종 출력: 스팟별 순위표(언급도 상/중/하|시간대|한줄팁),
제외 추천과 이유, 일몰 시각표, 출처 URL.
```

### 트랙 C: 공식 운영정보 실무 검증 ⭐가장 중요

```
{날짜} {도시들} 여행의 실무 정보를 공식 사이트 기준 최신으로 검증해줘.

일정 전제: {요일 포함 도시별 체류 일정}

검증 항목:
1~N. {주요 명소별}: 개장시간(요일별!), 티켓 종류/가격,
   시간지정 예약 필요 여부, 예약 오픈 시기와 매진 속도
- 요일 이슈: 일요일 미사(성당), 월요일 휴관(박물관), 국경일
- 도시별 관광객 교통패스 최신 가격
- 공항↔시내 이동 수단과 요금

최종 출력: 항목별 "확인된 사실 | 우리 일정에 미치는 영향 | 권장 조치(예약 시점)"
+ 예약 우선순위 표 (오늘 날짜 기준 "지금 즉시/N주 내/아무 때나") + 출처 URL.
가격은 현지통화+원화 병기.
```

> 실전 효과: 이 트랙에서 "빈 48시간권 2026년 폐지", "성 비투스 대성당 일요일 12시 개장", "국회의사당 영어투어 2~4주 전 매진"을 잡아냈습니다. 가이드북/블로그만 믿으면 틀리는 정보들입니다.

## 4. STEP 3 — 계획서 HTML 제작 프롬프트

```
리서치 결과를 반영해서 여행 계획서 index.html을 만들어줘/개선해줘.

UI 요구사항:
- 한 번에 날짜별로 제대로 보고, 길찾고, 선택할 수 있도록
- 상단 고정 날짜 칩 네비게이션 (D1~DN, 개요, 패키지 비교)
- 날짜별: 타임라인(시간·설명·이동수단) + 번호 지도(Leaflet) +
  연속 일정 구간마다 구글맵 길찾기 버튼 자동 생성
- 선택지 카드는 클릭으로 선택 저장(localStorage), 예약 체크리스트도 체크 저장
- 예약 우선순위는 오늘 날짜 기준 긴급도 순 + 공식 예약 링크 버튼
- 단일 파일(CDN만), 서버 불필요

데이터 규칙:
- 일정은 HTML 직접 작성 말고 JS 데이터 배열(DAYS) → 렌더 함수로 생성
  (지도 핀·타임라인·길찾기가 한 소스에서 나와 불일치 원천 차단)
- 확정 정보(항공/기차/숙소)는 저장된 값 그대로, 배지로 "확정" 표시
- 작성 후 <script> 추출해서 node --check로 문법 검증
```

구현 세부(데이터 모델, 길찾기 URL 규칙, lazy 지도 초기화 등)는 `travel-plan-builder/SKILL.md` 참고.

## 5. STEP 4 — 후기 링크 실접속 검증 프롬프트

### 신규 수집

```
여행 계획서에 넣을 네이버 블로그 후기 링크를 수집해줘.

1. 아래 각 스팟에 대해 네이버 블로그 후기 링크 1개씩 (가급적 2024~2026 최신)
2. 찾은 URL은 반드시 실제 접속해서 ①열리는지 ②주제가 맞는지 확인한 것만 포함
3. m.blog.naver.com 모바일 URL로 통일
4. 정말 못 찾으면 "못 찾음" — 무리하게 아무 링크나 넣지 마

대상: {스팟 목록}
출력: 표 (스팟 | URL | 글 제목 | 작성시기 | 검증✅)
```

### 기존 링크 전수 검증

```
아래 URL들을 각각 접속해서 ①정상(삭제/비공개 아님) ②주제 일치를 확인해줘.
깨졌거나 주제가 다르면 같은 주제의 최신 후기를 찾아 검증 후 대체 URL 제시.
{URL 목록}
```

> 실전 효과: 15개 중 4개가 "제목은 맞는데 내용이 다른 글"이었고(프라하 야경투어 링크가 실제론 부다페스트 글), 조사 중 **페트르진 푸니쿨라 2025~2026 운행중단** 같은 일정에 영향 주는 정보도 발견됨.
> ⚠️ 네이버 블로그는 AI 크롤러를 차단하므로, WebFetch 실패 시 curl(브라우저 UA) 등으로 HTTP 200 + 제목/키워드 확인하도록 안내하세요.

## 6. STEP 5 — 마이리얼트립 투어 조회 (공식 MCP)

별도 API키 불필요. `myrealtrip-search` 스킬의 래퍼 스크립트 사용:

```bash
export PYTHONIOENCODING=utf-8   # Windows 필수 (cp949 인코딩 오류 방지)
S=".agents/skills/myrealtrip-search/scripts/myrealtrip_mcp.py"

# ① 검색
python $S call searchTnas --arg query="프라하 야경" --arg perPage=5

# ② 우리 날짜에 예약 가능한지 확인 (핵심!)
python $S call getTnaOptions --json \
  '{"gid":"3434345","url":"https://experiences.myrealtrip.com/products/3434345","selectedDate":"2026-08-08"}'
# → isRequestedDateAvailable / options[].salePrice 로 판단
```

프롬프트로 시키려면:

```
마이리얼트립 MCP로 각 날짜·도시에 맞는 투어를 검색하고,
우리 일정 날짜에 실제 예약 가능한지(getTnaOptions)까지 확인해서
가능한 상품만 날짜별 추천 카드로 계획서에 넣어줘.
동선·시간(오전/오후/야간)이 기존 일정과 겹치지 않는지도 고려해.
예약 불가 상품은 제외하고 그 사실을 기록에 남겨.
```

> 실전 효과: 평점 4.9·후기 2,546개짜리 인기 투어가 정작 우리 날짜(8/11) 예약 불가 → 조회 없이 넣었으면 낭패.

## 8. 자주 겪는 문제와 해결

| 문제 | 해결 |
|---|---|
| 여행사 사이트(SPA)가 안 열림 | 공식 블로그·검색 스니펫·유사 패키지 일정표로 대체 |
| 네이버 블로그 크롤러 차단 | 브라우저 UA로 HTTP 요청 → 200 + og:title/키워드 확인 |
| Windows에서 파이썬 한글 깨짐 | `export PYTHONIOENCODING=utf-8` |
| 숨긴 탭의 Leaflet 지도 깨짐 | 탭 첫 표시 때 초기화(lazy) + 재표시 때 `invalidateSize()` |
| 같은 정보가 문서 곳곳에서 어긋남 | 데이터(JS 배열) → 렌더 함수 구조로 단일 소스화 |
| 가격/운영시간이 소스마다 다름 | 공식 사이트 우선, 상충 시 "결제화면 기준 확인" 표기 + 조사일 명시 |
